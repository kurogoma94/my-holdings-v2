# 機能企画書：大阪ディープ居酒屋ルーレット

## 1. 背景・目的

大阪には無数の居酒屋があるが、「今日どこ行こう？」と一人飲みで迷うことは多い。
本アプリは、**ルーレットの偶然性**で新しいお店との出会いを生み出す。
Takahiroが実際に足を運んだ"ディープな"居酒屋だけを厳選収録し、**ハズレなしのルーレット体験**を提供する。

SNS事業部（Instagram @takahiroshimizu94）との連動により、アプリへの集客とSNSフォロワー増の相乗効果を狙う。

---

## 2. 対象ユーザー（ペルソナ）

| 項目 | 内容 |
|------|------|
| **メインターゲット** | 大阪在住の一人飲み好きの男性（25〜45歳） |
| **サブターゲット** | 大阪出張・観光の飲み好き層 |
| **ニーズ** | 「今日の一軒」を手軽に決めたい / 知らない名店を開拓したい |
| **流入経路** | Instagram投稿 → プロフィールリンク → アプリDL or Webアクセス |

#### タブ1: 🎰 ルーレット（メイン画面）

```
┌──────────────────────────┐
│                          │
│   大阪ディープ居酒屋      │
│      ルーレット           │
│                          │
│  ┌────┬────┬────┐        │
│  │すべて│エリア│ジャンル│   │  ← フィルタータブ（3つ）
│  └────┴────┴────┘        │
│                          │
│  [エリア/ジャンル選択UI]   │  ← 「エリア」「ジャンル」選択時のみ表示
│                          │
│     ┌──────────┐         │
│     │          │         │
│     │ ルーレット │         │  ← 回転アニメーション付き
│     │  ホイール  │         │
```

- **すべて**: 全店舗からランダムに1件を選出

- **ジャンル**: ジャンルを選択 → そのジャンル内からランダムに選出

### 3.2 店舗データの構成

| フィールド | 型 | ソース | 必須 |
|-----------|-----|--------|------|
| `id` | string | 自動生成 | ✅ |
| `name` | string | 手動入力 | ✅ |
| `area` | string | 手動入力（天満、難波、鶴橋 等） | ✅ |
| `genre` | string | 手動入力（立ち飲み、焼き鳥、居酒屋 等） | ✅ |
| `budget_min` | number | 手動入力 | ✅ |
| `budget_max` | number | 手動入力 | - |
| `comment` | string | 手動入力（Takahiroの一言） | ✅ |
| `rating` | number (1-5) | 手動入力 | ✅ |
| `photo_url` | string | 手動 or Google Places API | ✅ |
| `google_maps_url` | string | Google Maps API で自動生成 | ✅ |
| `place_id` | string | Google Places API | - |
| `address` | string | Google Places API で自動取得 | - |
| `phone` | string | Google Places API で自動取得 | - |
| `opening_hours` | object | Google Places API で自動取得 | - |
| `google_rating` | number | Google Places API で自動取得 | - |
| `is_active` | boolean | 自動（閉店管理） | ✅ |
| `created_at` | datetime | 自動 | ✅ |

**データ入力フロー:**
1. Takahiroが「店名」「エリア」「ジャンル」「予算」「コメント」「評価」を手動入力
2. 店名 + エリアから Google Places API で `place_id` を検索・紐付け
3. `place_id` に基づき、住所・電話・営業時間・写真・Google評価を自動取得
4. 将来的にユーザー投稿機能を追加する余地を残す（`submitted_by` フィールド等を拡張可能に設計）

## 4. 技術スタック

### MVP（Phase 1）
| レイヤー | 技術 | 理由 |
|---------|------|------|
| **フロントエンド** | React Native (Expo) | iOS/Android同時対発、Webエクスポートも可能 |
| **バックエンド** | Firebase (Firestore + Auth) | サーバーレスで運用コスト最小、スケーラブル |
| **API** | Google Places API / Maps SDK | 店舗の詳細情報・地図表示 |
| **画像ストレージ** | Firebase Storage | 店舗写真の保管 |
| **アナリティクス** | Firebase Analytics | MAU等のKPI計測 |

### 将来（Phase 2以降）
| 項目 | 技術 |
|------|------|
| Web版 | Expo for Web or Next.js（React Native Web） |
| ユーザー認証 | Firebase Auth（Google/Apple ログイン） |
| プッシュ通知 | Firebase Cloud Messaging |

## 5. マネタイズ戦略（提案）

### 結論
→ **Phase 1は無料で配布し、ユーザー数を最優先で獲得。Phase 2以降で段階的に収益化する**のがおすすめ。

### Phase 1（リリース〜MAU 500人）：無料 + ブランディング
- **収益**: ¥0（投資フェーズ）
- **目的**: ユーザー獲得 & Instagram連動でフォロワー増
- **施策**: SNSでのシェア機能を徹底的に作り込む（「今日のお店」を自動的にストーリーズに投稿できる等）

### Phase 2（MAU 500〜2,000人）：広告 + アフィリエイト
| 手法 | 内容 | 想定月収 |
|------|------|---------|
| **AdMob広告** | ルーレット結果画面にインタースティシャル広告（3回に1回程度） | ¥5,000〜20,000 |
| **予約アフィリエイト** | 「ホットペッパーで予約する」ボタンで送客。1予約あたり報酬。 | ¥3,000〜15,000 |
| **Instagramとの相互送客** | アプリ→Insta、Insta→アプリでPR案件の単価を上げる | 間接効果 |

### Phase 3（MAU 2,000人〜）：店舗PR + プレミアム
| 手法 | 内容 | 想定月収 |
|------|------|---------|
| **店舗PR枠** | 居酒屋オーナーが月額課金で「おすすめ店舗」に掲載 | ¥10,000〜/店 |
| **プレミアムプラン** | 広告非表示 + お気に入り保存数無制限 + 限定フィルター | ¥300/月 |
| **EC連動** | アプリ内で「この店の味を自宅で再現」→ EC事業部の業務用調味料への送客 | 間接効果 |

> **💡 特にEC連動は他社にない差別化ポイント**。「この店で使ってるラードはこれ！」みたいな展開ができれば、EC事業部との相乗効果が生まれる。

## 8. リスクと対策

| リスク | 影響度 | 対策 |
|--------|--------|------|
| 店舗データが少なく魅力に欠ける | 高 | MVP時点で最低20件を確保。Instagram投稿済みの店舗を優先的に登録 |
| Google Places API のコスト超過 | 中 | APIコールをキャッシュし、1店舗につき1回のみフェッチ。月間無料枠内で運用 |
| ストア審査でリジェクト | 中 | Apple審査ガイドラインに準拠したUI設計。最低限のコンテンツ量を確保 |
| 閉店した店舗が残る | 低 | Google Places APIのステータスで定期チェック + ユーザー報告機能 |
